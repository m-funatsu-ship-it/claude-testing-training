# トレーニング概要

ATRS（航空券予約システム）を題材に、AIを活用したテスト設計スキルを実践的に習得するトレーニングの概要と運用情報。受講者・メンター双方の参照資料。

---

## 目的

**AIを道具として使いこなし、テスト設計を自律的に進められるようになること。**

---

## 進め方

- **AIがドラフトを作り、人間が直す**サイクルの繰り返し
- AIとの「壁打ち」を中心に、手を動かしながら実践の中で覚える
- Gitによる成果物管理を、課題の進行と並行して習得する

## AIツール利用における情報セキュリティ

**一度送信した情報は削除できない。**  
AIツールへの入力・GitHubへのpush・画面共有のいずれも、操作した瞬間に情報はサーバーに送信され、自分では制御できなくなる。研修中だけでなく、案件参画後も同様に適用される。

### 判断基準

> **「これは公開していい情報か？」—— 一秒で判断できないなら入力しない。**

### 入力してはいけない情報

| カテゴリ | 具体例 |
|---|---|
| 個人情報 | 氏名・住所・電話番号・メールアドレス・マイナンバー等 |
| 業務上の機密情報 | 顧客情報・社内ドキュメント・未公開仕様・売上データ等 |
| 認証情報 | パスワード・APIキー・トークン・SSH秘密鍵等 |
| 参画先の内部情報 | 案件のソースコード・DBスキーマ・インフラ構成等 |

**なぜ取り返しがつかないか：**  
「案件の仕様書をそのまま貼って質問した」という行為が、顧客との秘密保持契約（NDA）違反になり得る。Claude Code / ChatGPT等に入力したデータはプロバイダのサーバーに送信され、削除を要求しても完全な消去は保証されない。エラーログや設定ファイルにもパス・バージョン・環境情報が含まれることがある。**最小限の情報に絞ってから質問する。**

### Git / GitHubへのpush時の注意

> **実際に起きること：** APIキーを含むファイルをGitHubのpublicリポジトリにpushした直後、自動スキャンボットに検知され数分〜数時間以内に悪用されるケースが報告されている。AWSのアクセスキーが漏洩した場合、一晩で数十万円の請求が発生した事例もある。

- push前に `git diff` で差分を目視確認する
- `.env`・`*.pem`・`secrets/` は必ず `.gitignore` に登録する
- **誤ってpushしてしまった場合：** まずキーを無効化（ローテーション）し、すぐにメンターに報告する。履歴から削除しても漏洩は取り消せないと心得る

### 案件参画後のツール利用

- **参画初日**に「AIツールの利用可否」をチームリーダーまたはPMに確認する。承認なく導入しない
- 個人のSaaSアカウント（無料プラン）を業務に使わない。無料プランは学習データへの利用を許可している場合が多い
- **画面共有・スクリーンショット時も同様。** 顧客データ・社内情報が映り込んでいないか確認してから共有する

### カリキュラム詳細

#### 課題1：同値分割法・境界値分析でテスト条件を識別する

**目的：** 同値分割法で入力値を同値クラス（同値パーティション）に分類してテスト条件を識別する。境界値分析で同値クラスの境界値とその直前・直後の値を対象にテストする。両技法の基礎を実例で身につける。

**使用：** ATRS仕様書・Claude Code

---

#### 課題2：デシジョンテーブルでテストを設計する

**目的：** 複数の条件の組み合わせをデシジョンテーブルとして整理し、テストケースを設計する。

**使用：** ATRS仕様書・Claude Code

---

#### 課題3：PostmanでATRS APIをテストする

**目的：** PostmanからATRSのAPIにリクエストを送り、レスポンスを確認するAPIテストの基礎を習得する。

**使用：** Postman・ATRS正常版

---

#### 課題4：チケット予約APIのテストを設計・実行する

**目的：** Claude Codeと壁打ちしながら、ATRSのチケット予約APIに対するテストケースを設計し、Postmanで実行する。

**使用：** Postman・ATRS正常版・Claude Code

---

#### 課題5：ログイン機能の状態遷移図を作る

**目的：** ログイン機能の振る舞いを状態遷移図としてモデル化し、テストの観点を可視化する。

**使用：** ATRS仕様書・Claude Code・Mermaid（VS Codeプレビュー）

---

#### 課題6：シーケンス図でテストモデルを作る

**目的：** 予約フローをシーケンス図として表現し、正常系・異常系の分岐をテストモデルとして整理する。

**使用：** ATRS仕様書・Claude Code・Mermaid

---

#### 課題7：PlaywrightでE2E自動テストを動かす

**目的：** Claude Codeが生成したPlaywrightのテストコードを実行し、E2E自動テストの基本的な流れを体験する。

**使用：** Playwright・ATRS正常版・Claude Code

---

#### 課題8：PlaywrightでAPIテストを自動化する

**目的：** PlaywrightをAPIテストにも応用し、テスト自動化の対象をAPI層まで広げる。

**使用：** Playwright・ATRS正常版・Claude Code

---

#### 課題9：AI統合演習（テスト設計→自動化→不具合報告）

**目的：** 課題1〜8で学んだすべての要素を組み合わせ、AIを活用して「テスト設計→テストコード生成→実行→不具合報告」を一気通貫で体験する。課題1〜8の集大成。

**手順概要：**
1. シーケンス図・テスト条件をAIに生成させる（テスト設計）
2. テストコードをAIに生成させ、実行・修正する（自動化）
3. ATRSバグ版で不具合を発見し、不具合レポートを作成する（報告）
4. 成果物をGitでコミット・プルリクエストを作成する（成果物管理）

**使用：** Playwright・ATRS正常版・ATRSバグ版・Claude Code・Git / GitHub

---

#### 課題EX-1：Linuxコマンドの基礎（WSL2）

**対象：** 現場でLinux環境（WSL2を含む）を使う受講者のみ。全員必須ではない。

**目的：** WSL2のUbuntu環境で基本的なLinuxコマンドを習得する。現場のサーバー環境への対応力を養う。

**習得コマンド例：** `pwd` / `ls` / `cd` / `mkdir` / `touch` / `rm` / `cat` / `grep` / `ps` 等

---

#### 課題EX-2：curlでWebサービスの応答を確認する（WSL2）

**対象：** 課題EX-1実施者のみ。

**目的：** WSL2環境からcurlコマンドでHTTPリクエストを送信し、レスポンスを確認する。Postmanで行っていた操作をCLIで体験する。

### リスク対策

| リスク | 対策 |
|---|---|
| Claude Codeの操作でつまずく | エラーもClaude Codeに聞く練習として活用する |
| AIの出力をそのまま使ってしまう | メンターが「なぜこの出力を修正するか」を一緒に考える |
| テスト技法の理解が追いつかない | 理論先行にせず、課題1〜2の実例を使って体感的に学ぶ |
| Claude Codeの使用量が上限に達する | Anthropicの利用プランのアップグレードを検討する |
