## Step 1：正常系のPOSTリクエストを送る

まず、APIに正しいリクエストを送って「201 Created」が返ることを確認します。  
Step 2以降のテスト設計に入る前に、**APIが動いていることを手元で確かめる**のが目的です。

### APIの情報

```
HTTPメソッド：POST
エンドポイント：http://localhost:8082/atrs/api/v1/ticket
Content-Type：application/json
```

### リクエストボディ

以下のJSONをコピーし、2か所を書き換えてください。

- `flightName`：課題3の手順4でメモした便名
- `depDate`：課題3の手順3で使った日付（`yyyy-MM-dd` 形式に変換して入力）

> **`fareTypeCd` について：** `"OW"`（片道運賃）は片道フライトで広く使われる値です。正常系のリクエストで400が返ってきた場合は、課題3の空席照会レスポンスの `fareTypes` にある値を確認して使用してください。

```json
{
  "repFamilyName": "デンデン",
  "repGivenName": "タロウ",
  "repAge": 30,
  "repGender": "M",
  "repTel1": "03",
  "repTel2": "1234",
  "repTel3": "5678",
  "repMail": "test@example.com",
  "flightType": "OW",
  "selectFlightResourceList": [
    {
      "depDate": "ここに日付を入力（例：2026-05-15）",
      "flightName": "ここに便名を入力",
      "boardingClassCd": "N",
      "fareTypeCd": "OW"
    }
  ],
  "passengerResourceList": [
    {
      "familyName": "デンデン",
      "givenName": "タロウ",
      "age": 30,
      "gender": "M"
    }
  ]
}
```

### 操作手順

1. HTTPメソッドのドロップダウンから **POST** を選択する
2. URLフィールドに `http://localhost:8082/atrs/api/v1/ticket` を貼り付ける
3. **[Body]** タブをクリックする
4. **[raw]** を選択し、右のドロップダウンを **JSON** に変更する（これで `Content-Type: application/json` ヘッダーが自動で設定されます）
5. 上のJSONを貼り付け、`depDate` と `flightName` を書き換える
6. **[Send]** をクリックする

**確認チェックリスト：**

- [ ] ステータスコードが **201** であることを確認した
- [ ] レスポンスボディに `reserveNo`（予約番号）が含まれていることを確認した

> **GETとPOSTの違い：** GETはURLにパラメータを付けて送りますが、POSTはリクエストボディにJSONを入れて送ります。  
> 成功したときのステータスコードも「200 OK」ではなく「**201 Created**」です。  
> 「新しいリソースが作られた」という意味を持ちます。

## Step 3：Postmanで異常系のテストを実行する

Step 2で識別したテスト条件をカバーするテストケースを、観点ごとに1件ずつ選んで実行します。  
Step 1で送ったリクエストのリクエストボディを1か所だけ変えて、**[Send]** を押してください。  
各テストケースが終わったら、変更箇所を元に戻してから次に進みます。

> **ヒント：** 元の正常系のリクエストボディをテキストエディタなどにコピーしておくと、各テストケースの実行のたびに参照できて便利です。

### 実行するテストケース

| T2 | `repFamilyName` | `"アイウエオカキクケコア"`（11文字） | 400 |
| T3 | `repAge` | `17`（18歳未満） | 409 |
| T4 | `repTel3` | `"567"`（3桁） | 400 |

**確認チェックリスト：**

- [ ] T1：ステータスコードが **400** であることを確認した
- [ ] T2：ステータスコードが **400** であることを確認した
- [ ] T3：ステータスコードが **409** であることを確認した
- [ ] T4：ステータスコードが **400** であることを確認した

> **気づきポイント：** T3（年齢17歳）が409になったことを確認できましたか？  
> 課題2のデシジョンテーブルで識別した「予約代表者が18歳未満 → 予約不可」というルールが、  
> APIの実際の動作としても確認できました。  
> **テスト設計と実行がつながった**という体験です。  
> また、入力バリデーション違反（T1・T2・T4・T5）は **400 Bad Request**、ビジネスルール違反（T3）は **409 Conflict** と、エラーの種類によってステータスコードが異なることも確認できました。
