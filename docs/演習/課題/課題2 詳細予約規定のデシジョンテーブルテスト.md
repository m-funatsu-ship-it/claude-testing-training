# 課題2：詳細予約規定のデシジョンテーブルテストを Claude Code で作成しよう

**対象フェーズ：** Phase 2 Step 1 — 仕様書を読み込ませてテストケースを生成する  
**所要時間：** 40〜60 分（Git 作業含む）  
**難易度：** ★★☆（課題1の応用・新しい技法を体験）

---

## この課題で学ぶこと

課題1では「同値分割法」と「境界値分析」を使ってテスト条件を識別しました。

この課題では、**複数の条件が組み合わさって結果が決まるルール**を扱います。  
このような場合に有効な技法が**デシジョンテーブルテスト（決定表テスト）**です。

Claude Code を使ってデシジョンテーブルを生成し、組み合わせの漏れがないかをレビューする体験をします。

---

## デシジョンテーブルテストとは

複数の条件（YES/NO）の組み合わせと、それに対応する結果（アクション）を表にまとめたものです。

```
例：「傘を持つ」という意思決定

| 条件                 | ルール1 | ルール2 | ルール3 | ルール4 |
|---------------------|--------|--------|--------|--------|
| 雨が降っているか？     |   Y    |   Y    |   N    |   N    |
| 傘を持っているか？     |   Y    |   N    |   Y    |   N    |
| 傘を借りる           |        |   ✓    |        |        |
| そのまま出かける       |   ✓    |        |   ✓    |   ✓    |
```

条件の「Y/N が関係ない（どちらでも結果が変わらない）」場合は `-`（ダッシュ）で表します。  
これを**「関係なし」条件**と呼びます。

---

## テストベース（今回の対象範囲）

以下のビジネスルールのみを対象とします。  
ファイル：`docs\ATRS\テストベース\ATRS 要件定義書 Markdown版.md`  
セクション：**A04-02 ビジネスルール_フライトチケット予約規定**  
項目：**詳細予約規定について**

---

### 対象ルール

```
搭乗者・予約代表者は以下の条件を満たす必要がある。

条件1. 予約代表者の年齢は 18 歳以上であること。
条件2. 運賃種別がレディース割の場合、搭乗者の性別はすべて女性であること。
条件3. 運賃種別がグループ割の場合、搭乗者の合計数が 3 名以上であること。
        ※ グループ割の利用可能最少人数は3名（運賃一覧 表3より）
```

## 作業手順

### Step 1：ビジネスルールを自分で読む（10分）

対象の3つの条件を読んで、以下を考えてみましょう。  
まだ書き出さなくてよいです。

> **考えるポイント：**
> - 条件はいくつあるか？
> - 各条件は独立しているか？（他の条件に関係なく YES/NO が決まるか？）
> - 条件と条件の間に「この条件が成り立つときだけ、次の条件が意味を持つ」という関係はあるか？
> - 「予約が受け付けられない」のはどんな組み合わせのときか？

---

### Step 2：Claude Code に初回プロンプトを送る（10分）

以下のプロンプトを参考にして Claude Code に指示を送ってみましょう。

---

**プロンプト例（初回）：**

```
以下のビジネスルールについて、デシジョンテーブルテスト（決定表テスト）を作成してください。

条件と結果（予約可能かどうか）をMarkdownの表形式でまとめてください。
条件が関係しない（どちらでも結果が変わらない）場合は「-」で表してください。
最後に「このテーブルで何件のテストケースが必要か」も教えてください。

---
【ビジネスルール：詳細予約規定】

搭乗者・予約代表者は以下の条件を満たす必要がある。

条件1. 予約代表者の年齢は 18 歳以上であること。
条件2. 運賃種別がレディース割の場合、搭乗者の性別はすべて女性であること。
条件3. 運賃種別がグループ割の場合、搭乗者の合計数が 3 名以上であること。
        （グループ割の利用可能最少人数は3名）

※ レディース割とグループ割を同時に選択することはできない。
---
```

Claude Code からデシジョンテーブルが出力されたら、内容を確認します。

---

### Step 3：出力をレビューする（10分）

Claude Code が出力したデシジョンテーブルに対して、以下の**チェックリスト**で確認してください。

#### チェックリスト

- [ ] **条件の個数**
  - 今回のビジネスルールで意味を持つ条件が全部入っているか？
  - 逆に、余計な条件（ビジネスルールに書かれていないもの）が入っていないか？

- [ ] **予約できないパターン**
  - 「予約代表者が18歳未満」→ 予約不可、のルールが入っているか？
  - 「レディース割 かつ 男性搭乗者がいる」→ 予約不可、のルールが入っているか？
  - 「グループ割 かつ 搭乗者が2名以下」→ 予約不可、のルールが入っているか？

- [ ] **「関係なし」条件の扱い**
  - 「レディース割でない場合」は「搭乗者全員が女性か」は関係ないため `-` になっているか？
  - 「グループ割でない場合」は「搭乗者が3名以上か」は関係ないため `-` になっているか？
  - 「予約代表者が18歳未満」のルールでは、他の条件は `-` になっているか？

- [ ] **ルールの網羅性**
  - 「レディース割でも グループ割でもない通常運賃」のケースは入っているか？
  - 全パターンを通じて「予約できる」ケースと「できない」ケースの両方があるか？

---

### Step 4：追加プロンプトで補う（10分）

Step 3 で漏れを見つけたら、Claude Code に追加指示を出してみましょう。

---

**プロンプト例（追加指示 その1）：**

```
ありがとうございます。
以下の点を確認・修正してください。

1. 「予約代表者が18歳未満」のルールでは、他の条件（レディース割・グループ割・性別・人数）は
   結果に影響しないので「-」として1つのルールにまとめてください。
2. 「通常運賃（レディース割でもグループ割でもない）」かつ「予約代表者18歳以上」の
   ケースが入っているか確認してください。なければ追加してください。
3. 修正後のデシジョンテーブル全体を再出力してください。
```

---

**プロンプト例（追加指示 その2）：**

```
このデシジョンテーブルをテストケースとして実行する場合、
各ルールに対して「入力値の具体例」を1つずつ追加してください。
（例：年齢=17、運賃種別=片道運賃、搭乗者=男性1名 など）
```

---

### Step 5：最終的なデシジョンテーブルをまとめる（5分）

Claude Code の出力を確認し、自分で整理・修正を加えて最終版のデシジョンテーブルを完成させます。  
自分でレビューして「正しい」と確認できた行だけを残してください。

## 注意事項

- 今回の対象は「詳細予約規定」の3条件のみです。合計金額の算出や支払い期限は対象外です。
- 「レディース割とグループ割は同時選択できない」はプロンプトに明示する必要があります。  
  書かないと Claude Code がすべての組み合わせを列挙してしまう場合があります。
- デシジョンテーブルの「ルール数」は、「関係なし」条件（`-`）の扱い方によって変わります。  
  Claude Code が何件と答えたか、自分の判断と一致するか確認してみましょう。

## Step 6：Git で成果物を管理する（自力でやってみよう）

デシジョンテーブルが完成したら、Git で記録して GitHub にプッシュします。  
**今回はメンターのサポートなしで、自力でやってみましょう。**

1. ブランチを作成する（`feature/exercise-2`）
2. `git status` で変更ファイルを確認する
3. ファイルをステージングする（ファイル名を指定して）
4. コミットメッセージを書いてコミットする
5. プッシュする（初回なので `-u` オプションが必要）

```bash
git push -u origin feature/exercise-2
```

6. ブラウザで GitHub を開き、`feature/exercise-2` ブランチにファイルが届いていることを確認する
7. PR（プルリクエスト）を作成して `main` にマージする
8. マージ済みブランチを削除する
9. `main` ブランチに3ファイルが反映されていることを確認する

詰まったときは [Git操作手順書.md](Git操作手順書.md) を参照してください。
