# 課題1：運賃算出ロジックのテスト条件を Claude Code で識別しよう

**対象フェーズ：** Phase 2 Step 1 — 仕様書を読み込ませてテストケースを生成する  
**所要時間：** 30〜50 分  
**難易度：** ★☆☆（Claude Code 初回体験）

---

## 課題の目的

この課題では、ATRS の要件定義書に書かれたビジネスルールを Claude Code に渡し、  
**テスト条件の一覧を自動生成させる**体験をします。

AI が出力した結果をそのまま正解と受け取るのではなく、  
**自分の目でレビューして漏れを補う**というサイクルを体験することがゴールです。

---

## テストベース（今回の対象範囲）

以下のビジネスルールのみを対象とします。  
ファイル：`docs\ATRS\テストベース\ATRS 要件定義書 Markdown版.md`  
セクション：**A04-01 ビジネスルール_空席状況表示規定**

対象とするルールは以下の 4 点です（運賃種別の表3は今回の対象外）。

---

### ルール1：運賃の算出方法

```
運賃 = 基本運賃 - 運賃種別ごとの割引額
※ 運賃の 100 円未満は切り上げとする。
```

---

### ルール2：基本運賃の算出方法

```
基本運賃 = （区間の基本料金 + 搭乗クラスの加算料金） × ピーク時期料金積算比率(%)
```

---

### ルール3：搭乗クラスの加算料金（表1）

| 搭乗クラスタイプ | 加算料金（円） |
|---|---|
| 一般席 | 0 |
| 特別席 | 5,000 |

---

### ルール4：ピーク時期別の積算比率（表2）

| 時期区分 | 時期 | 積算比率（%） |
|---|---|---|
| ピーク1 | 1月1日〜1月5日 / 3月20日〜31日 / 8月8日〜18日 / 12月26日〜31日 | 140 |
| ピーク2 | 7月18日〜8月7日 / 8月19日〜31日 / 12月19日〜25日 / 3月13日〜19日 | 125 |
| 通常時 | 上記以外 | 100 |

---

## 成果物

以下の形式のMarkdown表を作成してください。

| # | 観点 | テスト条件 | 入力値例 | 期待結果 |
|---|---|---|---|---|
| 1 | 搭乗クラス | 一般席の場合、加算料金は0円 | 搭乗クラス＝一般席 | 加算料金：0円 |
| 2 | … | … | … | … |

> **用語補足：**  
> 「テスト条件」はテストすべき側面（何をテストするか）を表します。「入力値例」と「期待結果」はその条件を具体的に示すための例示です。  
> JSTQB では「テスト条件を識別する（テスト分析）→ テストケースを設計する（テスト設計）」という順序で進めます。この表は両者を1行にまとめた**ハイレベルテストケース**に近い形式であり、実際のテスト実行には具体的な入力値・手順を定めたテストケースへの展開が必要になる場合があります。

---

## 作業手順

### Step 1：ビジネスルールを自分で読む（10分）

上記の「ルール1〜4」を読んで、**どんな入力パターンが考えられるか**を頭の中で整理します。  
まだ書き出さなくてよいです。「どんな境界がありそうか？」という視点で読んでください。

> **ヒント：** 「搭乗クラスが2種類ある」「ピーク期間が複数ある」「切り上げが発生する条件は何か」などを意識してみましょう。

---

### Step 2：Claude Code に初回プロンプトを送る（10分）

VS Code で Claude Code を開き、以下のプロンプトを参考にしながら指示を送ってみましょう。  
**そのままコピーして使ってもよいし、自分の言葉に変えてもかまいません。**

---

**プロンプト例（初回）：**

```
以下のビジネスルールからテスト条件を識別してください。
同値分割法と境界値分析の観点でテスト条件を洗い出し、Markdown の表形式で出力してください。
表の列は「#」「観点」「テスト条件」「入力値例」「期待結果」としてください。

---
【ビジネスルール】

■ 運賃の算出方法
運賃 = 基本運賃 - 運賃種別ごとの割引額
※ 運賃の 100 円未満は切り上げとする。

■ 基本運賃の算出方法
基本運賃 = （区間の基本料金 + 搭乗クラスの加算料金） × ピーク時期料金積算比率(%)

■ 搭乗クラスの加算料金
| 搭乗クラスタイプ | 加算料金（円） |
|---|---|
| 一般席 | 0 |
| 特別席 | 5,000 |

■ ピーク時期別の積算比率
| 時期区分 | 時期 | 積算比率（%） |
|---|---|---|
| ピーク1 | 1月1日〜1月5日 / 3月20日〜31日 / 8月8日〜18日 / 12月26日〜31日 | 140 |
| ピーク2 | 7月18日〜8月7日 / 8月19日〜31日 / 12月19日〜25日 / 3月13日〜19日 | 125 |
| 通常時 | 上記以外 | 100 |
---
```

Claude Code から表が出力されたら、内容を確認します。

---

### Step 3：出力をレビューする（10分）

Claude Code が出力した表に対して、以下の**チェックリスト**で抜けを確認してください。

#### チェックリスト

- [ ] **搭乗クラス（同値分割法）**
  - 一般席（加算料金 0 円）のテスト条件が入っているか？
  - 特別席（加算料金 5,000 円）のテスト条件が入っているか？

- [ ] **ピーク時期（同値分割法）**
  - ピーク1・ピーク2・通常時の 3 区分がそれぞれテスト条件として入っているか？

- [ ] **ピーク時期（境界値分析）**
  - ピーク期間の**開始日前日と開始日当日**が入っているか？（例：12月25日 vs 12月26日）
  - ピーク期間の**終了日当日と翌日**が入っているか？（例：1月5日 vs 1月6日）
  - ピーク1とピーク2が**隣接している時期**の境界が入っているか？（例：3月19日 vs 3月20日）

- [ ] **切り上げルール**
  - 「100円未満が発生するケース」と「100円未満が発生しないケース」が入っているか？

---

### Step 4：追加プロンプトで補う（10分）

Step 3 で漏れを見つけたら、Claude Code に追加指示を出してみましょう。

---

**プロンプト例（追加指示）：**

```
ありがとうございます。
以下の観点が不足していたので、追加のテスト条件を出力してください。

1. ピーク時期の境界値（ピーク期間の開始日前日と開始日当日、終了日当日と翌日）
2. ピーク1とピーク2が隣接している日付の境界（例：3月19日はピーク2、3月20日はピーク1）
3. 運賃の100円未満切り上げが発生するケースと発生しないケース

上記を先ほどの表に追記した形で、全体を再出力してください。
```

出力された表を、Step 3 のチェックリストで再確認してください。

---

### Step 5：最終的な表をまとめる（5分）

Claude Code の出力を確認し、自分で整理・修正を加えて最終版の表を完成させます。  
自分でレビューして「正しい」と確認できた行だけを残してください。

> **ポイント：** Claude Code の出力をそのまま提出するのではなく、  
> **「自分がレビューして承認した」状態にすること**が重要です。

## 注意事項

- Claude Code に渡すのは**上記「テストベース」に記載したルール1〜4のみ**です。  
  要件定義書の全文を渡す必要はありません。
- 今回は**テスト条件の識別**が目的です。実際のテスト実行やコード作成は行いません。
- ピーク期間の境界は複数あります（ピーク1が4期間、ピーク2が4期間）。  
  すべての境界を網羅できたか、最後に確認してみましょう。

## Step 6：Git で成果物を管理する（メンターと一緒に）

テスト条件一覧が完成したら、Git でバージョン管理します。  
**今回はメンターと一緒に操作します。画面を見せながら進めましょう。**

操作の流れ：

1. ブランチを作成する（`feature/exercise-1`）
2. `git status` で変更ファイルを確認する
3. ファイルをステージングする（ファイル名を指定して）
4. コミットメッセージを書いてコミットする
5. GitHub にプッシュする
6. ブラウザで GitHub を開き、ファイルが届いていることを確認する
7. PR（プルリクエスト）を作成して `main` にマージする
8. マージ済みブランチを削除する
9. `main` ブランチに反映されていることを確認する

詳細な手順はこちらを参照：[Git操作手順書.md](Git操作手順書.md)
