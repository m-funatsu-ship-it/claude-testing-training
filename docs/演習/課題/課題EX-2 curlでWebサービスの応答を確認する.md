# 課題EX-2：curl で Web サービスの応答を確認する

> **番外編について：** この課題はメインの演習シーケンス（課題1〜8）とは独立しています。Linux 操作に不安がある受講者向けの補足演習です。

**所要時間：** 20〜30 分  
**対象：** Linux コマンドの操作に不安がある受講者  
**前提：**
- [課題EX-1](<課題EX-1 Linuxコマンドの基礎を習得する.md>) を完了していること（nginx がインストール済みであること）

---

## この課題の目的

現場では「デプロイ後にサービスが正常に応答しているか確認する」作業があります。  
この課題のゴールは「`curl` コマンドを使って、Web サービスの応答内容とステータスコードを確認できること」です。

---

## 準備：Ubuntu のシェルに入る

VS Code の「ターミナル」→「新しいターミナル」を開き、以下のコマンドを実行して Ubuntu のシェルに切り替えます。

```
wsl
```

プロンプトが `username@hostname:/mnt/...` のような形式に変わったら、続けてホームディレクトリに移動します。

```
cd ~
```

プロンプトが `username@hostname:~$` のような形式に変わったら準備完了です。

---

## Step 1：curl で HTTP リクエストを送る

`curl` は URL に対して HTTP リクエストを送り、応答を表示するコマンドです。ブラウザのアドレスバーに URL を入力する操作を、コマンドで行うイメージです。

### 手順1：nginx が起動していることを確認する

```
sudo systemctl status nginx
```

**確認チェックリスト：**

- [ ] `Active: active (running)` と表示された

> **パスワードを求められた場合：** WSL セットアップ時に設定したパスワードを入力してください（入力中は画面に表示されません）。

> **出力が長くてページャーが起動した場合：** 内容を確認したら `q` を押して終了してください。

> **nginx が起動していない場合：** 以下のコマンドで起動してください。
> ```
> sudo systemctl start nginx
> ```

---

### 手順2：curl で応答を確認する

```
curl http://localhost
```

**確認チェックリスト：**

- [ ] HTML のコードが大量に表示された

> **メモ：** `localhost` は「自分自身のサーバー」を指すアドレスです。ブラウザで `http://localhost` を開いたときと同じ応答が、コマンドで取得できています。

---

### 手順3：HTTP ステータスコードだけを確認する

```
curl -I http://localhost
```

**確認チェックリスト：**

- [ ] `HTTP/1.1 200 OK` のような行が表示された

> **メモ：** `-I` オプションは HTTP HEAD メソッドでリクエストを送り、レスポンスヘッダーのみを表示します。先頭の `HTTP/1.1 200 OK` の数字（ステータスコード）がサービスの状態を表します。
>
> | ステータスコード | 意味 |
> |---|---|
> | `200 OK` | 正常に応答している |
> | `404 Not Found` | URL が存在しない |
> | `500 Internal Server Error` | サーバー側でエラーが起きている |
>
> テスト作業では「期待するステータスコードが返ってきているか」を確認することがよくあります。

---

## Step 2：結果をファイルに保存する

コマンドの出力を画面に表示するだけでなく、ファイルに保存することができます。テスト作業では「確認したときの状態を記録として残す」場面があります。

### 手順4：curl の結果をファイルに保存する

```
curl http://localhost > response.html
cat response.html
```

**確認チェックリスト：**

- [ ] `cat` の出力に HTML のコードが表示された

> **メモ：** `>` は「コマンドの出力をファイルに書き込む」記号です（リダイレクト）。手順2で画面に表示された内容が、そのままファイルに保存されます。

> **プログレスバーが表示された場合：** `curl` は実行中に進捗情報（`% Total ...` のような行）を表示することがあります。これはエラーではありません。`cat` で HTML が表示されていれば正常に保存されています。

---

### 手順5：保存したファイルの行数を確認する

```
wc -l response.html
```

**確認チェックリスト：**

- [ ] 数字（行数）が表示された

> **気づきポイント：** EX-1 で使った `wc -l` をここでも活用できます。行数が表示されれば、`>` によってファイルに内容が保存されたことが確認できます。

---

## Step 3：応答がない場合を確認する

存在しないポートへリクエストを送ることで、エラー時の挙動を体験します。

### 手順6：応答がない URL にリクエストを送る

```
curl http://localhost:9999
```

**確認チェックリスト：**

- [ ] 接続できないことを示すエラーメッセージが表示された（`Connection refused` や `Could not connect to server` など）

> **メモ：** ポート `9999` には何も動いていないため、即座に接続エラーが返ります。表示されるメッセージは環境によって異なりますが、いずれも「接続先が見つからない」という意味です。テスト現場では「サービスが起動していないのにリクエストを投げてしまった」というケースでよく見るエラーです。

## 振り返り

以下の問いに自分なりの言葉で答えてみてください。

1. `curl http://localhost` と `curl -I http://localhost` の違いは何ですか？
2. ステータスコード `200`・`404`・`500` はそれぞれどんな状態を意味しますか？
3. コマンドの結果を `>` でファイルに保存するのは、どんな場面で役立ちますか？

## 演習を終えるとき

演習が終わったら、以下のコマンドを実行して WSL シェルを終了してからターミナルを閉じてください。

```
exit
```

> **注意：** ウィンドウの閉じるボタンを直接押さずに、必ず `exit` で終了する習慣をつけてください。SSH でサーバーに接続して作業する場合も同様で、`exit` による正常終了はサーバー管理の基本作法です。
