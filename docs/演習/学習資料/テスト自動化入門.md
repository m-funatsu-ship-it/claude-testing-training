## この資料の目的

「テスト自動化」と聞くと、プログラミングの知識が必要な難しい技術に聞こえるかもしれません。  
この資料では、**テスト自動化とは何か・なぜ使うのか・何ができて何ができないのか**を整理します。

ゴールは「自動化ツールを使う理由と限界を、自分の言葉で説明できるようになること」です。

---

## Part 1：手動テストの限界

### 繰り返しコストの問題

課題1〜6で行ったテストを思い出してください。

- ログイン画面で会員番号・パスワードを入力する
- ボタンをクリックする
- 画面の表示を目で確認する
- 結果を記録する

1回のテストとして見れば、それほど大変な作業ではありません。  
では、これを**毎週リリースのたびに実行する**としたらどうでしょうか。

```
ソフトウェアの変更
 → ログイン機能に影響していないか確認したい
 → 先週と同じテストをまた実行する
 → 来週の変更でもまた同じテストを実行する
 → ……
```

新機能を追加するたびに「以前の機能が壊れていないか」を確認するテストを**リグレッションテスト（回帰テスト）** と呼びます。  
リリース頻度が高いシステムでは、このリグレッションテストのコストが積み上がっていきます。

### 人間が苦手なこと

手動テストには、人間ならではの強みがあります。  
「画面のレイアウトが不自然に見える」「この文言はユーザーに伝わりにくい」といった判断は、人間にしかできません。

一方で、人間が苦手なこともあります。

| 人間が苦手なこと | 具体例 |
|---|---|
| 全く同じ手順を何百回も繰り返すこと | テストのたびに同じ入力値を打ち込む |
| 疲れても精度を落とさないこと | 50件目のテストで確認を見落とす |
| 複数の環境で同時に実行すること | 複数ブラウザで同じ操作を並行して行う |

テスト自動化は、こうした「人間が苦手な繰り返し作業」を機械に委ねる技術です。

## Part 2：テスト自動化とは何か

### 「手順書をコードに書き直す」という考え方

手動テストでは、テスト手順書に沿って人間が操作します。

```
【手動テストの手順書】
1. ブラウザで http://localhost:8082/atrs/auth/login?form を開く
2. 会員番号欄に「0000000001」を入力する
3. パスワード欄に「aaaaa11111」を入力する
4. ログインボタンをクリックする
5. TOP画面に遷移したことを確認する
6. 「ようこそ」という文字が表示されていることを確認する
```

テスト自動化とは、**この手順書をプログラムが読める言葉で書き直したもの**です。  
プログラムは書かれた手順を何度でも、疲れずに、同じ速さで実行します。

```
【自動テストのコード（イメージ）】
ブラウザを開く
ログイン画面に移動する
会員番号欄に「0000000001」を入力する
パスワード欄に「aaaaa11111」を入力する
ログインボタンをクリックする
TOP画面に遷移したことを確認する        ← アサーション（失敗するとフェイル）
「ようこそ」という文字が表示されていることを確認する ← アサーション（失敗するとフェイル）
```

テストが「パス」か「フェイル」かは、コードに書いた「確認する」ステップ（**アサーション**）の結果で決まります。  
アサーションが1つでも失敗したら、その時点でテストはフェイルになります。

### テスト自動化のメリット

| メリット | 内容 |
|---|---|
| 繰り返し実行コストの削減 | 一度書いたテストは、手動で実行するたびの人件費をかけずに繰り返し実行できる |
| 実行の効率化 | 人の手を介さず夜間・並列での実行が可能なため、テスト全体の実行時間を短縮できる |
| 安定性 | 毎回同じ手順を同じ精度で実行する |
| 記録 | パス・フェイルの結果が自動でレポートに残る |

### テスト自動化のデメリット（限界）

自動化は万能ではありません。自動化が苦手なことも理解しておく必要があります。

| デメリット | 内容 |
|---|---|
| 初期コストがかかる | テストコードの作成と実行環境の構築に時間がかかる |
| 継続的なメンテナンスが必要 | 機能変更のたびにテストコードの修正が必要になる |
| 変更に弱い | 画面のHTML構造が変わるとテストが失敗しやすい |
| 「おかしさ」に気づけない | レイアウトのずれ・文章のニュアンスは判断できない |
| 実行していないことは確認できない | コードに書いていない確認は行われない |

> **ポイント：** テスト自動化は「同じテストを繰り返し実行すること」が得意です。  
> 「初めて見る画面の問題を発見すること」は人間の方が得意です。  
> 自動化と手動テストは対立するものではなく、**役割分担**です。

## Part 3：テストピラミッド

ソフトウェアテストには複数の種類があり、よく「テストピラミッド」という図で整理されます。

```
             ┌───────────┐
             │ E2Eテスト  │  ← 少ない（コストが高い）
           ──┼───────────┼──
          │   APIテスト   │
        ────┼─────────────┼────
       │    ユニットテスト   │  ← 多い（コストが低い）
       └─────────────────────┘
```

E2Eテストは実行に時間がかかり、HTML構造の変更で壊れやすくメンテナンスコストも高いため、費用対効果を考えて件数を絞るのが一般的です。

下層のユニットテストほど高速・低コストで数を増やしやすく、上層に向かうほど件数を絞ります。

> **補足：** 一般的なテストピラミッドでは中間層を「統合テスト（Integration Test）」と呼ぶことが多く、APIテストはその代表的な形態の一つです。この資料では学習で使ったPostmanによるテストに合わせて「APIテスト」と表記しています。

### 各テストの役割

| テストの種類 | 確認すること | 使ったツール・フレームワーク |
|---|---|---|
| ユニットテスト | 関数・メソッド単体の動作 | （今回は未経験） |
| APIテスト | サーバーとのデータのやり取り | Postman（課題3・4） |
| E2Eテスト | ユーザーが画面を操作する流れ全体 | Playwright（次の課題） |

### APIテストとE2Eテストの違い

課題3・4でPostmanを使ったAPIテストを体験しました。  
E2EテストはAPIテストとどう違うのでしょうか。

```
【APIテスト（Postman）】
テスター → HTTPリクエストを直接送信 → サーバー → レスポンスを確認

【E2Eテスト（Playwright）】
テスター → ブラウザを操作するコードを実行 → ブラウザ → 画面の表示を確認
                                              ↓
                                           サーバーへのリクエストは
                                           ブラウザが自動的に行う
```

| 比較 | APIテスト（Postman） | E2Eテスト（Playwright） |
|---|---|---|
| 操作対象 | APIエンドポイント（URL） | ブラウザの画面 |
| 確認内容 | JSONのレスポンス | 画面の表示・遷移 |
| ブラウザ | 不要 | 必要（Playwrightが制御） |
| 速度 | 速い | 遅い（ブラウザを動かすため） |
| 向いている用途 | ロジックの検証 | ユーザー操作の流れの検証 |

> **E2E（End-to-End）** とは「端から端まで」という意味です。  
> ユーザーがブラウザを開いてからシステムのレスポンスが返るまで、  
> **システム全体を通した流れ**をテストすることを指します。

## Part 4：自動化に向いているテスト・向いていないテストとは

テストを自動化すれば工数が減るように思えますが、**自動化に向いているテスト**と**向いていないテスト**があります。

### 自動化に向いているテスト

- **リグレッションテスト**：繰り返し実行するもの。リリースのたびに同じテストを実行する場合
- **正常系の基本的なフロー**：仕様が安定していて、頻繁に変わらないもの
- **複数パターンのシナリオテスト**：正常系・異常系など、同じ画面操作を複数のデータパターンで確認したいもの

### 自動化に向いていないテスト

- **探索的テスト**：テスト設計と実行を同時並行で行うアプローチ。前のテスト結果から学びながら次の操作を決めるため、事前に手順を固定できない
- **ユーザビリティ確認**：「見やすいか」「操作しやすいか」という主観的な評価
- **一度しか実行しないテスト**：コストをかけてコードを書いても、実行機会が少ない

> **ポイント：** 「全部自動化する」という発想は現実的ではありません。  
> 「何を自動化すると一番コストパフォーマンスが高いか」という視点が重要です。

## まとめ

| 項目 | 要点 |
|---|---|
| テスト自動化とは | 手動テストの手順書をプログラムが読める言葉で書き直したもの |
| 得意なこと | 同じテストの繰り返し実行（リグレッションテスト） |
| 苦手なこと | 「おかしさ」の発見、初見の問題の発見 |
| テストピラミッドとは | ユニットテスト・APIテスト・E2Eテストの3層でテストの種類を整理する考え方 |
| APIテストとE2Eテストの違い | APIテストはHTTPリクエストでサーバーを直接確認する。E2Eテストはブラウザを操作して画面の表示を確認する |
| E2Eの意味 | ユーザーの操作を起点に、システム全体を通じた流れをテストすること |
| 自動化に向くテスト | 仕様が安定していて繰り返し実行するもの（リグレッションテストなど） |
| 自動化に向かないテスト | 探索的テスト・ユーザビリティ確認・一度しか実行しないテスト |

---

## 用語集

| 用語 | 意味 |
|---|---|
| テスト自動化 | 主にテストの実行をプログラムで行うこと |
| E2Eテスト（End-to-Endテスト） | ユーザーの操作を起点に、システム全体を通した流れをテストすること |
| リグレッションテスト（回帰テスト） | 変更後に既存の機能が壊れていないかを確認するテスト |
| アサーション | テストコードの中で「このような状態になっているはずだ」と確認するステップ |
| テストピラミッド | ユニット・API・E2Eの3層でテストを整理する考え方 |
| パス（Pass） | テストが成功した状態（アサーションがすべて満たされた） |
| フェイル（Fail） | テストが失敗した状態（アサーションの1つ以上が満たされなかった） |
