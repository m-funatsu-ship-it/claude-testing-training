# Git 操作手順書

演習課題の成果物を Git で記録し、GitHub に送信するための手順書です。  
課題1ではメンターと一緒に、課題2では自力でこの手順を実行します。

---

## Git とは

**ファイルの変更履歴を記録するツール**です。

テスト条件一覧を作成・修正するたびに「いつ・何を変えたか」が記録され、後から見返したり元の状態に戻したりできます。

| Git がない場合 | Git がある場合 |
|---|---|
| `テスト条件_最終版.md` | `テスト条件.md`（1つのファイル） |
| `テスト条件_最終版2.md` | 変更履歴は Git が管理してくれる |
| `テスト条件_本当の最終版.md` | いつでも過去の状態に戻せる |

テスト成果物を Git で管理すると、「このテストケースはなぜ追加されたのか」「いつ修正されたのか」が追跡できるようになります。現場でも同様の運用が一般的です。

---

## GitHub とは

**Git の記録をクラウドに保存・共有するサービス**です。

Git はあくまで「自分のPC上での記録ツール」です。GitHub に送信（push）することで、他のメンバーと共有したり、PCが壊れてもデータが失われないようにできます。

```
自分の PC                          GitHub（クラウド）
┌──────────────────┐              ┌──────────────────┐
│ ローカルリポジトリ │── push ───→ │  リモートリポジトリ │
│                  │← pull  ───  │  （共有・バックアップ）│
└──────────────────┘              └──────────────────┘
```

| | Git | GitHub |
|---|---|---|
| 何か | 変更履歴を記録するツール | 記録をクラウドで共有するサービス |
| 場所 | 自分のPC内 | インターネット上 |
| オフラインで使えるか | 使える | 使えない |

---

## Git の基本構造

```
① 作業ツリー           ② ステージング           ③ ローカルリポジトリ      ④ リモート（GitHub）
（編集中のファイル）    （コミット対象の選択）      （自分のPCに記録済み）     （クラウドに送信済み）

    ─── git add ──→    ─── git commit ──→        ─── git push ──→
```

| 操作 | 何をするか | GitHub に届くか |
|---|---|---|
| `git add` | コミットしたいファイルをステージングに並べる | 届かない |
| `git commit` | ステージングのファイルをローカルリポジトリに記録する | 届かない |
| `git push` | ローカルリポジトリの記録をリモート（GitHub）に送信する | **届く** |

> `git commit` だけでは GitHub にはまだ届きません。`git push` して初めて反映されます。

## 初回セットアップ（リポジトリがない場合）

**はじめて Git を使う場合**は、このセクションを先に実施します。  
セットアップ済みの場合は「手順」へ進んでください。

---

### A. ローカルリポジトリを初期化する

`Ctrl + Shift + P` でコマンドパレットを開き、`Git: Initialize Repository` を選択する。  
フォルダ選択が出たらプロジェクトのルートフォルダを選択する。

ソース管理パネルにファイル一覧が表示されれば成功。

---

### B. .gitignore を作成する

コミットしてはいけないファイル（自動生成物・機密情報など）を Git の管理対象から除外します。  
VS Code のターミナル（`` Ctrl + ` ``）で以下を実行します。

```powershell
@"
node_modules/
test-results/
.claude/
"@ | Out-File -FilePath .gitignore -Encoding utf8
```

> **なぜ除外するのか**
> - `node_modules/` ：ライブラリ群。大量のファイルがあり、コマンドで再生成できるためコミット不要。
> - `test-results/` ：テスト実行のたびに変わる自動生成物。
> - `.claude/` ：Claude Code の作業ファイル。個人環境の設定が含まれる場合がある。

> **警告について：** `warning: LF will be replaced by CRLF` が表示された場合はエラーではありません。  
> Windows の改行コードに関する通知であり、動作に影響しないため無視してかまいません。

---

### C. GitHub にリモートリポジトリを作成する

1. [github.com](https://github.com) にログインする
2. 右上の `+` → 「New repository」をクリックする
3. リポジトリ名を入力して「Create repository」をクリックする
4. **「Add a README file」のチェックは外す**（既存ファイルと競合するため）
5. 作成後のページに表示される HTTPS の URL をコピーしておく

---

### D. リモートリポジトリを登録する

ターミナルで以下を実行します（URL は C でコピーしたものに置き換える）。

```bash
git remote add origin https://github.com/<ユーザー名>/<リポジトリ名>.git
```

登録できたか確認します。

```bash
git remote -v
```

以下のように表示されれば成功です。

```
origin  https://github.com/<ユーザー名>/<リポジトリ名>.git (fetch)
origin  https://github.com/<ユーザー名>/<リポジトリ名>.git (push)
```

---

### E. 初回コミット＆プッシュ

```bash
git add .gitignore
git add docs/
git status                    # ステージング内容を確認する
git commit -m "feat: 演習課題の成果物とトレーニングドキュメントを追加"
git push -u origin main
```

> **初回のみ `docs/` をディレクトリごとステージングしています。**  
> 2回目以降（課題ごとの作業）はファイルを1つずつ指定してステージングします。詳細は「3. ファイルをステージングする」を参照してください。

GitHub のリポジトリページをブラウザで開き、ファイルが反映されていることを確認します。

## 課題ごとの Git 作業手順

初回セットアップ完了後、課題のたびに以下の手順を繰り返します。

### 1. ブランチを作成する

作業中のファイルは `main` ブランチに直接置かず、専用のブランチを作って作業します。

**VS Code GUI：**

1. 画面左下のブランチ名（例：`main`）をクリックする
2. 「＋ 新しいブランチを作成...」を選択する
3. ブランチ名を入力して Enter

**ターミナル（CLI）：**

```bash
git checkout -b feature/exercise-1
```

---

### 2. 変更内容を確認する

ステージングする前に、何が変わったかを必ず確認します。

**VS Code GUI：**

1. 左サイドバーの「ソース管理」アイコンをクリックする
2. 「変更」の一覧にファイルが表示されていることを確認する
3. ファイル名をクリックすると差分が右側に表示される

**ターミナル（CLI）：**

```bash
git status        # 変更されたファイルの一覧
git diff          # 具体的な差分の内容
```

---

### 3. ファイルをステージングする

`git add .`（全ファイル一括）は使いません。ファイルを1つずつ指定してステージングします。

**VS Code GUI：**

1. 「変更」の一覧でファイルにカーソルを合わせると `+` ボタンが表示される
2. `+` をクリックして「ステージングされた変更」に移動させる
3. 対象ファイルをすべてステージングする

**ターミナル（CLI）：**

```bash
git add "docs/演習課題/課題1 運賃算出のテスト条件識別.md"
git add "docs/演習課題/課題2 詳細予約規定のデシジョンテーブルテスト.md"
git status   # ステージングされたことを確認する
```

---

### 4. コミットする

コミットメッセージは「何をしたか」が分かるように書きます。  
先頭の `feat:`・`fix:` は操作の種類を示すプレフィックスです（`feat:` は機能追加、`fix:` は修正）。

**コミットメッセージの例：**

```
feat: 運賃算出・詳細予約規定のテスト条件一覧を追加

- 課題1：同値分割法・境界値分析で運賃算出のテスト条件を識別
- 課題2：デシジョンテーブルテストで詳細予約規定のテスト条件を識別
```

| 良い例 | 悪い例 |
|---|---|
| `feat: 運賃算出のテスト条件一覧を追加（同値分割・境界値）` | `課題1完了` |
| `fix: ピーク期間の境界値が漏れていたため追記` | `修正` |

**VS Code GUI：**

1. 「ソース管理」パネル上部のテキストボックスにコミットメッセージを入力する
2. `✓` ボタン（「コミット」）をクリックする

**ターミナル（CLI）：**

```bash
git commit -m "feat: 運賃算出・詳細予約規定のテスト条件一覧を追加"

# コミットされたことを確認する
git log --oneline
```

---

### 5. リモート（GitHub）にプッシュする

コミット直後、GitHub をブラウザで確認してみましょう。  
まだファイルが届いていないことを確認してからプッシュします。

**VS Code GUI：**

1. 「ソース管理」パネルの `...` メニューをクリックする
2. 「プッシュ」を選択する
3. 「ブランチを発行しますか？」が表示されたら「OK」をクリックする

**ターミナル（CLI）：**

```bash
# 初回は -u オプションをつける（リモートブランチとの追跡関係を設定する。以降は git push のみでよくなる）
git push -u origin feature/exercise-1
```

プッシュ後、GitHub をブラウザで再読み込みして、ファイルが届いていることを確認します。

確認できたら **「プルリクエスト（PR）を作成してマージする」** セクションへ進んでください。

---

## プルリクエスト（PR）を作成してマージする

プッシュ後、ブランチの変更を `main` に取り込むために PR を作成します。

> **ブラウザ操作が基本。** PR はコードレビューやコメントなど視覚的な操作が多いためブラウザが主流です。  
> ターミナルから操作したい場合は GitHub 公式 CLI ツール `gh` を使います（別途インストールが必要）。

---

### 1. PR を作成する

**ブラウザ：**

1. `https://github.com/<ユーザー名>/<リポジトリ名>` を開く
2. 画面上部に黄色いバナーで「`feature/xxx` had recent pushes — **Compare & pull request**」が表示される
3. 「Compare & pull request」をクリックする
4. タイトルにコミットメッセージを入力する（自動入力されている場合はそのままでOK）
5. 「Create pull request」をクリックする

> 「No conflicts with base branch / Merging can be performed automatically.」と表示されていれば問題ありません。

**ターミナル（CLI）：**

```bash
gh pr create --title "feat: 課題の成果物を追加" --base main
```

---

### 2. マージする

**ブラウザ：**

1. PR ページで「**Merge pull request**」をクリックする
2. 「**Confirm merge**」をクリックする
3. 「Pull request successfully merged and closed」が表示されれば成功

**ターミナル（CLI）：**

```bash
gh pr merge --merge
```

---

### 3. ブランチを削除する

マージ済みのブランチは不要になるため削除します。

**ブラウザ：**

1. 「**Delete branch**」をクリックする

**ターミナル（CLI）：**

```bash
# リモートブランチを削除する
git push origin --delete feature/exercise-1

# ローカルブランチを削除する
git branch -d feature/exercise-1
```

---

### 4. main ブランチへの反映を確認する

GitHub のブラウザ上で以下を確認します（VS Code 上ではなく、ブラウザでの操作です）。

1. リポジトリのトップページでブランチが「`main`」になっていることを確認する
2. `docs/演習課題/` フォルダを開き、コミットしたファイルが存在することを確認する

確認できたら、ローカルの `main` ブランチにリモートの変更を取り込みます。

```bash
git checkout main
git pull
```

## よくあるエラーと対処法

### `error: remote origin already exists`

**原因：** `git remote add origin` をすでに実行済み、または VS Code のコマンドパレットが部分的に登録してしまった場合に発生します。

**対処：** `add` の代わりに `set-url` で URL を上書きします。

```bash
git remote set-url origin https://github.com/<ユーザー名>/<リポジトリ名>.git
```

---

### VS Code の `Git: Add Remote` でURL入力欄が出てこない

**原因：** コマンドパレットの `Git: Add Remote` は、リモート名の入力後に URL 入力欄が表示されないことがあります。

**対処：** ターミナルで直接コマンドを実行します。

```bash
git remote add origin <URL>
# すでに origin が存在するエラーが出た場合
git remote set-url origin <URL>
```

---

### `warning: LF will be replaced by CRLF`

**原因：** Linux/Mac 形式の改行（LF）を Windows 形式（CRLF）に変換するという Git の通知です。

**対処：** エラーではないため無視してかまいません。動作に影響はありません。

---

## コマンドチートシート

```bash
# 状態確認
git status                              # 変更ファイルの一覧を確認する
git diff                                # 差分の内容を確認する
git log --oneline                       # コミット履歴を確認する
git remote -v                           # リモートの登録状況を確認する

# ブランチ操作
git checkout -b <ブランチ名>             # ブランチを作成して切り替える

# ステージング・コミット・プッシュ
git add <ファイルパス>                   # ファイルをステージングする
git commit -m "<メッセージ>"             # コミットする
git push -u origin <ブランチ名>          # リモートにプッシュする（初回）
git push                                # リモートにプッシュする（2回目以降）

# ブランチ削除
git branch -d <ブランチ名>              # ローカルブランチを削除する
git push origin --delete <ブランチ名>   # リモートブランチを削除する

# PR操作（gh コマンド：別途インストールが必要）
gh pr create --title "<タイトル>" --base main   # PRを作成する
gh pr merge --merge                             # PRをマージする

# 設定
git config --global core.quotepath false   # 日本語ファイル名を読みやすく表示する
```
