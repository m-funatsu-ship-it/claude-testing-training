## この資料の目的

この資料では、APIテストの基本概念とPostmanの基本操作を学びます。  
ゴールは「Postmanを使ってリクエストを送り、レスポンスを確認できる状態になること」です。

## Part 1：APIとは

### APIとは何か

**API（Application Programming Interface）** とは、ソフトウェア同士が情報をやり取りするための「窓口」です。

レストランに例えると、次のような関係になります。

```
① 客（ブラウザ・アプリ）  →  注文をウェイターに渡す  ← リクエスト
② ウェイター（API）       →  厨房に注文を伝える
③ 厨房（サーバー）        →  料理を作る
④ ウェイター（API）       →  料理を客に届ける        ← レスポンス
```

- **リクエスト**：クライアントがAPIに送る「要求」（①の注文）
- **レスポンス**：APIがクライアントに返す「応答」（④の料理）
- この一往復がAPIの基本的な通信の流れです

### なぜAPIテストをするのか

Webアプリは「画面（UI層）」と「バックエンドのAPI（サービス層）」を主な2層として構成されています。

| テストの種類 | 対象 | 確認すること |
|---|---|---|
| UIテスト | 画面 | ボタン・表示・操作の動作 |
| APIテスト | API | データの入出力・ビジネスルールの適用 |

UIテストだけでは「画面の問題」と「ロジックの問題」が混在し、不具合の原因特定が難しくなります。  
APIテストでサービス層を直接確認することで、**問題の切り分け**が容易になります。

## Part 2：HTTPの基礎知識

この資料で扱うWebアプリのAPIは、HTTP（HyperText Transfer Protocol）というプロトコル（通信の規約）で通信します。  
ここでは課題に必要な最低限の知識だけを押さえます。

### HTTPメソッド

リクエストには「何をしたいか」を示す **HTTPメソッド** を指定します。

| HTTPメソッド | 操作 | 例 |
|---|---|---|
| GET | データを取得する | 予約情報を参照する |
| POST | データを新規作成する | 新しい予約を登録する |
| PUT | データを更新する | 予約内容を変更する |
| DELETE | データを削除する | 予約をキャンセルする |

> 今回の課題では **GET** と **POST** を使います。PUT と DELETE は参考として掲載しています。

### ステータスコード

レスポンスには「処理の結果」を示す **ステータスコード**（3桁の数字）が含まれます。

| 範囲 | 意味 | 代表的なコード |
|---|---|---|
| 200番台 | 成功 | 200 OK、201 Created |
| 400番台 | クライアント側の誤り | 400 Bad Request、404 Not Found |
| 500番台 | サーバー側の問題 | 500 Internal Server Error |

### リクエストの構造

リクエストは主に以下の要素で構成されます。

| 要素 | 説明 | 例 |
|---|---|---|
| HTTPメソッド | 操作の種類 | GET、POST |
| エンドポイント | リクエスト先のURL（パスまで） | `https://example.com/api/flights` |
| クエリパラメータ | URLの末尾に `?` で繋いで付けるデータ。主にGETリクエストで検索条件の指定に使われる | `?date=2025-01-01` |
| ボディ | 送信するデータ（主にPOSTで使用） | JSON形式のデータ |

```
【GETリクエストの例】
HTTPメソッド：GET
エンドポイント：https://example.com/api/flights
クエリパラメータ：?date=2025-01-01

【POSTリクエストの例】
HTTPメソッド：POST
エンドポイント：https://example.com/api/reservations
ボディ：
{
  "name": "山田太郎",
  "date": "2025-01-01",
  "seat": "economy"
}
```

> **ヘッダー** という要素もありますが、今回の課題では使用しません。

## Part 3：Postmanのセットアップ

### Postmanとは

Postmanは、APIに対してリクエストを送信し、レスポンスを確認できるツールです。  
ブラウザのアドレスバーからはGETリクエストしか送れませんが、PostmanではPOSTなどのHTTPメソッドも画面から操作できます。

### インストール

VS Code の拡張機能マーケットプレイスから **Postman** 拡張機能をインストールします。

1. VS Code のサイドバーにある拡張機能アイコン（四角が4つ並んだアイコン）をクリックする
2. 検索ボックスに `Postman` と入力する
3. 発行元が **Postman** のものを選択し、**[インストール]** をクリックする

インストールが完了すると、VS Code のサイドバーに Postman のアイコンが追加されます。

### サインイン

Postman VS Code 拡張機能の使用には **Postmanアカウントへのサインイン** が必要です。

1. サイドバーの Postman アイコンをクリックする
2. **[Sign In]** をクリックするとブラウザが開くので、案内に従いサインインする

> アカウントをお持ちでない場合は、同じ画面から無料アカウントを作成できます。

### ワークスペースとコレクション

Postmanでリクエストを管理するために、2つの概念を理解しておきます。

| 概念 | 役割 |
|---|---|
| **ワークスペース** | 作業のまとまりを管理する場所。コレクションやリクエストを置く領域 |
| **コレクション** | 関連するリクエストをまとめるフォルダ |

**ワークスペースの作成手順：**

1. Postmanパネルで **[Create Workspace]** をクリックする
2. 種類を選択する（個人練習目的なら **Personal** または **Internal** でよい）
3. ワークスペース名を入力して **[Create]** をクリックする

**コレクションの作成手順：**

1. ワークスペースを開いた状態で **[Create Collection]** をクリックする
2. コレクション名を入力する

> **Postman Consoleについて：** PostmanパネルにはConsoleタブがありますが、これはリクエストの送受信ログを確認するデバッグ用のエリアです。リクエストの作成・送信はコレクション内のリクエスト画面から行います。

### 新規リクエストの作成

コレクションを作成したら、コレクション名の横の **[+]** をクリックしてリクエストを追加します。  
エディタエリアにリクエスト設定画面が開きます。

### 画面の構成

Postmanの主な操作エリアは3か所です。

| エリア | 役割 |
|---|---|
| ① リクエスト設定エリア | HTTPメソッド・エンドポイントを入力する。クエリパラメータは **Params** タブ、送信データは **Body** タブで設定する |
| ② [Send]ボタン | リクエストを送信する |
| ③ レスポンスエリア | ステータスコード・レスポンスの内容を確認する |

レスポンスエリアには以下のタブがあります。通常は **Pretty** タブで確認します。

| タブ | 表示内容 |
|---|---|
| Pretty | 整形されたJSON（通常はこちらを使う） |
| Raw | 整形なしの生テキスト |
| Preview | HTML形式のプレビュー |

## Part 4：JSONPlaceholderで動作確認

### JSONPlaceholderとは

[JSONPlaceholder](https://jsonplaceholder.typicode.com/) は、練習用に公開されているダミーAPIです。  
アカウント登録不要・APIキー不要で、すぐにリクエストを送れます。

> **注意：** JSONPlaceholderはテスト用のダミーAPIです。  
> POSTで送信したデータは実際には保存されません。  
> ステータスコードとレスポンスの形式を確認する練習として使用します。

### 練習手順とチェックリスト

以下の①〜③を順番に操作し、確認できたらチェックをつけてください。

---

#### ① GETリクエストを送る（正常系）

```
HTTPメソッド：GET
エンドポイント：https://jsonplaceholder.typicode.com/posts/1
```

**操作手順：**

1. HTTPメソッドのドロップダウンから **GET** を選択する
2. URLフィールドにエンドポイントを入力する
3. **[Send]** をクリックする
4. レスポンスエリアの **Status** でステータスコードを確認する
5. **Pretty** タブでレスポンスのJSONを確認する

**確認チェックリスト：**

- [ ] ステータスコードが **200** であることを確認した
- [ ] レスポンスのJSONに `id`・`title`・`body`・`userId` が含まれていることを確認した

---

#### ② GETで存在しないリソースを取得する（異常系）

```
HTTPメソッド：GET
エンドポイント：https://jsonplaceholder.typicode.com/posts/9999
```

**操作手順：** ①と同じ手順でエンドポイントのみ変更して送信する

**確認チェックリスト：**

- [ ] ステータスコードが **404** であることを確認した
- [ ] レスポンスが空の `{}` であることを確認した

> **気づきポイント：** 存在しないIDを指定すると404が返ります。  
> これはAPIテストにおける「異常系」テストのイメージです。  
> ①の正常系（200）と区別して覚えておきましょう。

---

#### ③ POSTリクエストをJSONボディ付きで送る

```
HTTPメソッド：POST
エンドポイント：https://jsonplaceholder.typicode.com/posts
```

送信するボディ（JSON）：

```json
{
  "title": "テスト投稿",
  "body": "Postmanの練習です",
  "userId": 1
}
```

**操作手順：**

1. HTTPメソッドのドロップダウンから **POST** を選択する
2. URLフィールドにエンドポイントを入力する
3. **Body** タブをクリックする
4. **raw** を選択し、右側のドロップダウンから **JSON** を選択する
5. 入力エリアに上記のJSONを入力する
6. **[Send]** をクリックする
7. レスポンスエリアでステータスコードとJSONを確認する

**確認チェックリスト：**

- [ ] ステータスコードが **201** であることを確認した（200ではなく201）
- [ ] レスポンスのJSONに `id` が含まれていることを確認した

> **気づきポイント：** 新規作成の成功は200ではなく201（Created）です。  
> GETの200と区別して覚えておきましょう。

## 用語集

| 用語 | 意味 |
|---|---|
| API（Application Programming Interface） | ソフトウェア同士が情報をやり取りするための窓口 |
| HTTP（HyperText Transfer Protocol） | Web上でデータを送受信するためのプロトコル |
| HTTPメソッド | リクエストで指定する操作の種類（GET・POST・PUT・DELETE） |
| エンドポイント | APIにアクセスするためのURL（パスまで。クエリパラメータは含まない） |
| リクエスト | クライアントからAPIへ送る要求 |
| レスポンス | APIからクライアントへ返す応答 |
| ステータスコード | レスポンスに含まれる3桁の数字。処理結果を示す |
| JSON（JavaScript Object Notation） | APIでデータを送受信する際に使われるデータ形式 |
| ボディ | リクエストやレスポンスに含まれるデータ本体 |
| ヘッダー | リクエストやレスポンスに含まれるメタ情報（認証情報・データ形式など） |
| クライアント | APIにリクエストを送る側（ブラウザ・アプリ・Postmanなど） |
| サーバー | リクエストを受け取り、レスポンスを返す側 |
| プロトコル | 通信の規約。データをどのような形式・手順でやり取りするかを定めたルール |
| ワークスペース | Postmanでリクエストやコレクションをまとめて管理する領域 |
| コレクション | Postmanで関連するリクエストをまとめる機能。フォルダのように整理できる |
