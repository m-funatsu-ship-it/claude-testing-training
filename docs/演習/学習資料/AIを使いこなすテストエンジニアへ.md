# AIを使いこなすテストエンジニアへ：現場に出る前の確認

**読了目安：約15分**  
**対象：トレーニングをすべて終えた受講者**

---

## この資料の目的

このトレーニングは、AIを道具として使いこなし、テスト設計を自律的に進められる状態を目指して設計されています。

この資料では、そのことを2つの観点から確認します。

1. **自分が身につけたことを、現場で使える形で整理する**
2. **現場に入る前に確認しておくべきことを把握する**

---

## Part 1：「AIを使って一人で動ける」とはどういう状態か

### このトレーニングの中心にあったサイクル

トレーニング全体を通じて繰り返してきたことは、一つのサイクルです。

```
AIがドラフトを作る → 人間が確認・修正する → 成果物になる
```

最後の統合演習ではこのサイクルを、テスト設計・テストコード生成・不具合レポートの3つで一気通貫で体験しました。

「AIを使って一人で動ける」とは、このサイクルの「確認・修正」を**自分で担える**ことです。

AIの出力を確認・修正せずにそのまま使っている状態では、ゴールに達したとは言えません。

### あなたが担う部分

| AIが担える部分 | あなたが担う部分 |
|---|---|
| テスト条件・テストケースの網羅的な案を出す | 仕様と照らして、抜けや誤りを判断する |
| テストコードの定型部分を生成する | 実行して確認し、アサーションの妥当性を判断する |
| 不具合レポートを構造化する | 重大度・影響範囲を判断する |

AIが生成したコードは、最初の実行でエラーになることがあります（実際に体験したように、空港ラベルの括弧が全角と半角で一致しないケースがありました）。実行エラーはすぐ気づけます。より注意が必要なのは、**テストが正常に通過するが、意図した検証をしていない**ケースです。AIは大量のコードを速く書けますが、「正しいことを確認しているかどうか」の判断はあなたに委ねられています。

## Part 2：AIの出力を正しく直すための判断軸

### なぜ人間がテスト設計の知識を持つ必要があるか

「AIが設計するなら、人間が技法を知らなくてもいいのでは？」という問いがあります。

答えは、**知らないと、AIの出力が正しいかどうか判断できないから**です。

AIは仕様を与えれば大量のテストケースを出します。しかし「大量」と「必要十分」は違います。

境界値分析の演習で作成したテストケースを振り返ってみましょう。「12歳以上が大人運賃」という仕様に対して：

- 11歳の乗客 → 子供運賃が適用される（境界の直前）
- 12歳の乗客 → 大人運賃が適用される（境界値そのもの）
- 13歳の乗客 → 大人運賃が適用される（境界の直後）

この3件は「12歳が子供か大人かの境界」という仕様の核心を突いています。AIにただ「テストケースを作って」と指示すれば、「東京→大阪の予約ができること」のような操作確認は出てきます。しかし、そうした指示だけでは**境界値のケースが漏れやすく**、境界値である11歳・12歳・13歳を必ずテストすべきだという判断は、**知識のある人間がレビューするか、的確な指示を出して初めて成立します。**

これらの技法は「AIの代わりに人間がやること」ではなく、「**AIの出力を正しく直すための判断軸**」です。

### 各判断軸の使いどころ

| 知識 | AIの出力で確認すること |
|---|---|
| 同値分割法・境界値分析 | 境界付近のケースが含まれているか。代表値だけで終わっていないか |
| デシジョンテーブル | 条件の組み合わせが論理的に網羅されているか |
| テストモデル（状態遷移図・シーケンス図） | 正常フロー・異常フローの両方がカバーされているか |

## Part 3：テストモデルが先にある理由

### まず、人間の理解のために作る

テストモデル（状態遷移図・シーケンス図）を先に作ることには、AIとは無関係の、本来の目的があります。

**① 仕様の構造を自分が理解するため**

仕様書はテキストで書かれています。それを図にしようとすると、「この遷移の後、システムはどの状態に戻るか」「この分岐、仕様書に書いてあったか」という問いが自然に生まれます。図を描く行為そのものが、仕様の読み込みになります。

**② 仕様の穴を早期に見つけるため**

テストケースを直接書き始めると、仕様の曖昧な部分を読み飛ばしたまま進んでしまうことがあります。モデルを先に作ると、つながらない遷移や定義されていない状態が図の上に現れます。これがテスト実行前に仕様の問題を発見する機会になります。

### さらに、AIへのインプットとしても機能する

テストモデルとして構造化した情報をAIに渡すと、出力の網羅性が上がります。

```
仕様書をそのまま渡す  → AIの出力が網羅性を欠きやすい
テストモデルを渡す    → AIが構造を読んで、カバーすべき分岐を特定しやすくなる
```

演習でAIが「予約成功と予約失敗の両方をカバーするテスト条件が必要」と正しく判断できたのは、シーケンス図の構造が渡されていたからです。

**テストモデルを作る能力は、仕様を正しく理解する力であり、AIに的確な指示を出す力でもあります。**

## Part 4：ツールの役割と使い分け

### テスト実行ツール

| ツール | 役割 |
|---|---|
| Postman | APIレベルでのリクエスト送信・検証 |
| Playwright | ブラウザ操作のE2E自動化 |

### APIテストとE2Eテストをどう使い分けるか

| 観点 | APIテスト（Postman） | E2Eテスト（Playwright） |
|---|---|---|
| 何を確認するか | サーバーのロジック・計算結果 | 画面の表示・ユーザー操作の流れ |
| 速さ・安定性 | 速い・安定 | 遅い・画面変更に弱い |
| 向いている場面 | 計算ロジック・バリデーションの検証 | 画面遷移・UIの統合動作 |

統合演習で発見した不具合（子供料金の計算ミス）はAPIテストでも検出できます。E2Eテストで発見したことの意味は、「**ユーザーが予約確認画面で実際に見る金額が間違っている**」という事実を、ユーザーと同じ経路で確認できた点にあります。

どちらが正しいという話ではなく、目的に応じて選択します。

## Part 5：現場に入る前の確認と、現場での習慣

### 確認：使えるAIは「お客様が指定したもの」だけ

現場では、使用できるAIツールはお客様が定めたものに限られます。

> **トレーニングで使ったAIツールを、現場でそのまま使うことはできません。**

これは習慣の問題でなく、情報セキュリティの問題です。

- テスト対象のシステム情報・仕様書は機密情報です
- お客様が許可していないAIツールに入力することは、情報漏洩になります
- 「便利だから」「似たようなものだから」という判断は許されません

**現場に入ったら最初に確認すること：「このプロジェクトで使用が許可されているAIツールは何か」**

### 習慣：成果物はGitで管理する

テスト設計書・テストコード・不具合レポートはすべてバージョン管理します。

「手元で完結させて後から共有する」ではなく、「作業のたびにコミット・プッシュ」が実務の基本です。

## Part 6：あなたが習得したスキルの全体像

```
AIを活用したテスト設計
├── 判断軸（AIの出力を正しく直すために）
│   ├── 同値分割法・境界値分析
│   └── デシジョンテーブル
├── テストモデリング（仕様を理解し、AIへの入力にするために）
│   ├── 状態遷移図
│   └── シーケンス図
├── ツール（テストを実行・検証するために）
│   ├── APIテスト（Postman）
│   └── E2Eテスト自動化（Playwright）
└── 成果物の管理（作ったものを共有・追跡するために）
    └── Git / GitHub / プルリクエスト
```

---

## まとめ

このトレーニングで身につけたことを一文で言えば：

> **AIにドラフトを作らせ、自分の判断で直して、成果物として管理できる。**

これが「AIを活用して一人でテストケースを設計できる」の実態です。

現場では、ツールが変わり、仕様が複雑になり、チームのやり方が異なります。それでもこのサイクルの構造は変わりません。

## 用語集

| 用語 | 意味 |
|---|---|
| 同値分割法 | 同じ動作をするグループに入力値を分類し、各グループから代表値でテストする技法 |
| 境界値分析 | 同値パーティションの境界付近の値を重点的にテストする技法 |
| デシジョンテーブル | 複数の条件の組み合わせに対する期待結果を表形式で整理する技法 |
| テストモデル | テスト設計に使いやすい形で仕様を可視化したもの（状態遷移図・シーケンス図など） |
| 重大度（severity） | 不具合がシステムやユーザーに与える影響の大きさ |
| E2Eテスト | ユーザーの操作を起点にシステム全体を通した流れをテストすること |
