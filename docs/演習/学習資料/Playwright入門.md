## Part 1：Playwrightとは

### ブラウザを操作するフレームワーク

Playwrightは、人間の代わりにブラウザを操作するフレームワークです。  
「URLを開く」「ボタンをクリックする」「テキストを入力する」「画面の表示を確認する」といった操作を、コードで記述して自動実行できます。

```
Playwrightのテストを実行すると…

1. Playwrightがブラウザを起動する（画面が出ることも・出ないこともある）
2. コードに書かれた手順を上から順番に実行する
3. 「確認する」ステップで、期待通りの状態かチェックする
4. 全手順が終わる（またはフェイルで中断する）と、結果（パス / フェイル）をレポートに記録する
5. ブラウザを閉じる
```

### 実行モード

Playwrightには2つの実行モードがあります。

| モード | 説明 | 使い分け |
|---|---|---|
| **ヘッドレスモード** | ブラウザが画面に表示されずバックグラウンドで動く | 通常の実行時（速い） |
| **ヘッドフルモード** | ブラウザが画面に表示されて操作の様子が見える | 動作確認・デバッグ時 |

> 今回のプロジェクトは通常時はヘッドレス、動作確認は `npm run test:headed` で実行します。

## Part 2：テストコードの構造

### テストファイルの見た目

このプロジェクトでは Playwright のテストファイルを `.spec.ts` という拡張子で命名しています。  
`tests/login.spec.ts` がその例です。

テストコードの大きな構造は次のようになっています。

```
テストグループ（test.describe）
 ├── 事前準備（test.beforeEach） ← 各テストの前に毎回実行される処理
 ├── テスト1（test）
 ├── テスト2（test）
 └── テスト3（test）
```

### 実際のコードと対応

`login.spec.ts` の一部を例に、各行が何をしているかを見てみます。

---

**テストグループの定義**

```typescript
test.describe('ログイン画面 (A101) - 状態遷移テスト', () => {
```

| コード | 意味 |
|---|---|
| `test.describe(...)` | テストをグループにまとめる。グループ名をレポートに表示する |
| `'ログイン画面 (A101) - 状態遷移テスト'` | グループの名前（日本語で書いてよい） |

---

**テスト実行前の共通処理**

```typescript
test.beforeEach(async ({ page }) => {
  await page.goto('auth/login?form');
  await expect(page).toHaveURL(/\/auth\/login/);
});
```

| コード | 意味 |
|---|---|
| `test.beforeEach(...)` | 各テストが始まる前に毎回実行される処理。ここでは「ログイン画面を開く」 |
| `page.goto(...)` | 指定したURLへ移動する |
| `expect(page).toHaveURL(...)` | 現在のURLが期待するものかを確認する（アサーション） |

---

**1つのテストケース**

```typescript
test('TC-L-001: 正常ログイン', async ({ page }) => {
  await submitLoginForm(page, '0000000001', 'aaaaa11111');
  await expect(page).toHaveURL(/\/ticket\/search/);
  await expect(page.getByText('ようこそ')).toBeVisible();
});
```

| コード | 手動テストで例えると |
|---|---|
| `test('TC-L-001: 正常ログイン', ...)` | テストケース番号と名前 |
| `submitLoginForm(page, '0000000001', 'aaaaa11111')` | 「会員番号とパスワードを入力してログインボタンをクリックする」 |
| `expect(page).toHaveURL(/\/ticket\/search/)` | 「期待結果：TOP画面に遷移すること」 |
| `expect(page.getByText('ようこそ')).toBeVisible()` | 「期待結果：『ようこそ』が表示されること」 |

## Part 3：3つの基本操作

Playwrightでよく使う操作は大きく3種類です。それぞれの意味を押さえておきましょう。

### ① 移動する

```typescript
await page.goto('auth/login?form');
```

ブラウザで指定したURLを開きます。  
手動テストの「ブラウザのアドレスバーにURLを入力してEnterキーを押す」に相当します。

---

### ② 操作する

```typescript
await page.fill('#membershipNumber', '0000000001');  // テキストを入力する
await page.click('#login-btn');                       // ボタンをクリックする
```

| 操作 | メソッド | 手動テストでの操作 |
|---|---|---|
| テキスト入力 | `page.fill(セレクタ, '値')` | 入力欄にキーボードで文字を入力する |
| クリック | `page.click(セレクタ)` | マウスでボタンや要素をクリックする |

> ※ 「セレクタ」とは何かは Part 4 で詳しく説明します。

---

### ③ 確認する（アサーション）

```typescript
await expect(page).toHaveURL(/\/ticket\/search/);
await expect(page.getByText('ようこそ')).toBeVisible();
await expect(page.locator('.alert')).toBeVisible();
```

| 確認内容 | コード | 手動テストでの確認 |
|---|---|---|
| URLが正しいか | `.toHaveURL(...)` | アドレスバーのURLを目視確認する |
| テキストが表示されているか | `.toBeVisible()` ＋ `getByText(...)` | 画面に「ようこそ」という文字が見えるか確認する |
| 要素が表示されているか | `.toBeVisible()` ＋ `locator(...)` | エラーメッセージのエリアが表示されているか確認する |

アサーションが期待通りでなければ、その時点でテストは**フェイル**になります。

## Part 4：セレクタとは

### 「どの要素を操作するか」を指定する方法

`page.fill('#membershipNumber', '0000000001')` の `#membershipNumber` の部分が**セレクタ**です。  
セレクタは「画面上のどの要素を操作するか」を指定する文字列です。

Webページの画面は、HTML（テキスト）で作られています。  
たとえばログイン画面の会員番号入力欄は、HTMLの中ではこのように書かれています。

```html
<input type="text" id="membershipNumber" name="membershipNumber">
```

Playwrightはこの `id="membershipNumber"` という情報をもとに、操作対象の要素を特定します。

### よく使う要素の指定方法

| 種別 | 例 | 意味 |
|---|---|---|
| `#id名`（セレクタ） | `#membershipNumber` | `id` 属性がその値の要素を指定する |
| `.クラス名`（セレクタ） | `.alert` | `class` 属性がその値の要素を指定する |
| `getByText(...)`（ロケーター） | `getByText('ようこそ')` | その文字列が含まれる要素を指定する |

> `#id名` / `.クラス名` は要素を特定する**文字列**（CSSセレクタ）です。  
> `getByText(...)` はPlaywright固有の**API**（ロケーター）で、戻り値は文字列ではなく要素への参照オブジェクトです。どちらも「操作対象の要素を特定する手段」として使います。

> **セレクタは自分で調べなくて大丈夫です。**  
> 課題7ではClaude Codeがセレクタを含むコード全体を生成します。  
> 「セレクタとは何かの概念」を知っていれば十分です。

## Part 5：テストの実行とレポート

### テストの実行コマンド

このプロジェクトでは、ターミナルで次のコマンドを実行してテストを動かします。

| コマンド | 動作 |
|---|---|
| `npm test` | すべてのテストをヘッドレスモードで実行する |
| `npm run test:headed` | すべてのテストをヘッドフルモードで実行する（ブラウザが表示される） |
| `npm run test:report` | 最後に実行したテストのHTMLレポートを開く |

### テスト結果の読み方

実行後、ターミナルに結果が表示されます。

```
✓  TC-L-001: 正常ログイン (1.2s)
✓  TC-L-002: クライアントエラー（会員番号 未入力） (0.8s)
✗  TC-L-003: クライアントエラー（パスワード 未入力） (2.1s)

3 tests, 1 failed
```

| 記号 | 意味 |
|---|---|
| `✓`（緑） | テストがパスした |
| `✗`（赤） | テストがフェイルした |

フェイルしたテストには、どのアサーションで失敗したかの詳細が表示されます。

### HTMLレポート

`npm run test:report` を実行するとブラウザでHTMLレポートが開きます。  
各テストケースの実行時間・成功・失敗が一覧で確認でき、フェイルしたテストの詳細も確認できます。

## まとめ

| 項目 | 要点 |
|---|---|
| Playwrightとは | ブラウザを自動操作してE2Eテストを実行するフレームワーク |
| テストファイルの構造 | `test.describe`（グループ）の中に複数の `test`（テストケース）が入る |
| 3つの基本操作 | ① 移動する（goto） / ② 操作する（fill・click） / ③ 確認する（expect） |
| セレクタとは | 画面上のどの要素を操作するかを指定する文字列 |
| アサーションとは | 「このような状態になっているはずだ」と確認するステップ。失敗するとフェイル |
| テスト設計との関係 | カバレッジアイテム（課題5）が、テストコードの1件以上に対応する |

## 用語集

| 用語 | 意味 |
|---|---|
| Playwright | Microsoftが開発したE2Eテスト自動化フレームワーク。Chromium（Chrome/Edgeなど）・Firefox・WebKit（Safariのベース）に対応 |
| `.spec.ts` | このプロジェクトで使用するテストファイルの拡張子。Playwright では慣習的に `.spec.ts` または `.test.ts` が使われることが多い |
| `test.describe` | テストケースをグループにまとめるブロック |
| `test.beforeEach` | 各テストの実行前に毎回呼ばれる共通処理 |
| `test` | 1つのテストケースを定義するブロック |
| `page` | ブラウザの1タブを表すオブジェクト。操作はすべてpageを通して行う |
| セレクタ | 操作対象のHTML要素を指定する文字列（例：`#id名`、`.クラス名`） |
| アサーション | テストコードの中で「このような状態になっているはずだ」と確認するステップ（`expect(...)` で書く） |
| ヘッドレスモード | ブラウザが画面に表示されずバックグラウンドで動作するモード |
| ヘッドフルモード | ブラウザが画面に表示され、操作の様子が見えるモード |
| パス（Pass） | テストが成功した状態（アサーションがすべて満たされた） |
| フェイル（Fail） | テストが失敗した状態（アサーションの1つ以上が満たされなかった） |
