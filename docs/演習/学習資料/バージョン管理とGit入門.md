# バージョン管理とGit入門

**所要時間：** 20〜30 分  
**対象：** Git・バージョン管理未経験の受講者

## この資料の目的

この資料では、バージョン管理の考え方とGitの基本概念を学びます。  
ゴールは「なぜGitを使うのか」「add・commit・pushが何をしているのか」を理解した状態で [Git操作手順書.md](Git操作手順書.md) に進めることです。

## Part 1：バージョン管理とは

### バージョン管理が必要な理由

テストドキュメントを作成していると、次のような状況が起きがちです。

```
テスト条件一覧.md
テスト条件一覧_修正.md
テスト条件一覧_最終.md
テスト条件一覧_最終2.md
テスト条件一覧_本当に最終.md
```

この管理方法には次の問題があります。

- どのファイルが最新かわからない
- いつ・何を変えたかわからない
- 昨日の状態に戻したくなっても、どれが「昨日の版」かわからない
- 複数人で作業すると上書きが起きる

**バージョン管理**とは、ファイルを1つのまま保ちながら「いつ・誰が・何を変えたか」を記録し、過去の状態にいつでも戻せるようにする仕組みです。

### Gitとは

**Git** は、バージョン管理を実現するツールです。  
ファイルの変更履歴を記録・管理します。具体的な記録の仕組みは Part 2 で説明します。

> **前提：** 演習環境にはGitがインストール済みです。  
> 確認したい場合は、ターミナルで `git --version` を実行してください。

## Part 2：Gitの基本概念

### リポジトリ

**リポジトリ** とは、Gitがファイルの変更履歴を保存している場所です。  
プロジェクトのフォルダ内に `.git` という隠しフォルダが作られ、そこにすべての履歴が記録されます。

### コミット

**コミット** とは、その時点のファイルの状態をスナップショット（ある時点の状態の完全な記録）として保存する操作です。  
写真を撮るようなイメージで、コミットするたびに「今の状態」が記録されます。

```
コミット1                コミット2                コミット3
「テスト条件を追加」  →  「境界値を追記」  →  「誤字を修正」
（1月10日 10:00）         （1月10日 14:30）       （1月11日 09:00）
```

過去のコミットに戻ることで、「昨日の状態に戻したい」という操作が可能になります。

### 変更をGitHubに届けるまでの流れ

Gitには4つの状態があります。変更内容はコマンドによってこの順番で記録されていきます。

```
① 作業ツリー         ② ステージング          ③ ローカルリポジトリ     ④ リモート（GitHub）
（編集中のファイル）   （コミット対象の選択）    （自分のPCに記録済み）   （クラウドに送信済み）

   ─── git add ──→    ─── git commit ──→       ─── git push ──→
```

| 操作 | 何をするか |
|---|---|
| `git add` | コミットしたいファイルをステージングに並べる |
| `git commit` | ステージングのファイルをローカルリポジトリに記録する |
| `git push` | ローカルリポジトリの記録をリモート（GitHub）に送信する |

> **重要：** `git commit` だけではGitHubにはまだ届きません。`git push` して初めて他のメンバーから見えるようになります。

## Part 3：ブランチとは

### ブランチが必要な理由

コミットの記録はmainブランチ上に直接積み重ねることもできますが、複数の作業を同時に進めると変更が混在してしまいます。

**ブランチ** とは、mainブランチから分岐した独立した変更の流れです。  
課題ごとに専用のブランチを作ることで、mainブランチを安全に保ちながら独立した作業ができます。

```
main               ─────────────────────────────────→（本流・完成した成果物を置く）
                        ↓ 分岐              ↑ マージ（取り込み）
feature/exercise-1 ──── コミット ──── コミット ─→（作業ブランチ）
```

### この課題でのブランチの使い方

| ブランチ | 役割 |
|---|---|
| `main` | レビュー済みの成果物を置く本流 |
| `feature/exercise-1` | 課題1の作業用 |
| `feature/exercise-2` | 課題2の作業用 |

作業ブランチで作業・コミットし、完成したらmainブランチにマージします。  
mainブランチに直接コミットしないのは、**「完成していない状態が本流に混ざらないようにするため」**です。

## Part 4：GitHubとプルリクエストとは

### GitHubとは

**GitHub** は、Gitの記録をクラウドに保存・共有するサービスです。  
GitはPC上のツール、GitHubはそれをインターネット上で保存・共有するサービスです。

| | Git | GitHub |
|---|---|---|
| 種類 | ツール | サービス |
| 場所 | 自分のPC内 | インターネット上 |
| オフラインで使えるか | 使える | 使えない |

### プルリクエスト（PR）とは

**プルリクエスト（PR）** とは、「作業ブランチの変更をmainブランチに取り込んでください」という申請です。

> **注意：** 名前が似ていますが、`git pull`（リモートからローカルリポジトリにデータを取得するコマンド）とは別の概念です。混同しないよう注意してください。

現場ではPRを通じてコードレビューが行われますが、この課題では**成果物を提出してmainブランチに反映するまでの一連の流れを体験すること**を目的とします。

```
① 作業ブランチ（feature/exercise-1）にコミットする
② GitHubにプッシュする
③ PRを作成する（「mainブランチに取り込んでください」という申請）
④ PRをマージする（mainブランチに反映される）
⑤ 作業ブランチを削除する
```

## 用語集

| 用語 | 意味 |
|---|---|
| バージョン管理 | ファイルの変更履歴を記録・管理する仕組み |
| Git | バージョン管理を実現するツール |
| GitHub | Gitの記録をクラウドで共有するサービス |
| リポジトリ | Gitが変更履歴を保存している場所 |
| コミット | その時点のファイルの状態を記録する操作、またはその記録 |
| スナップショット | ある時点のファイルの状態の完全な記録 |
| ブランチ | mainブランチから分岐した独立した変更の流れ |
| ステージング | コミットするファイルを選択する準備エリア |
| 作業ツリー（ワーキングツリー） | 実際に編集中のファイルが置かれている場所 |
| ローカルリポジトリ | 自分のPC上にあるリポジトリ |
| リモートリポジトリ | GitHub上にあるリポジトリ |
| プッシュ（push） | ローカルリポジトリの記録をリモート（GitHub）に送信すること |
| プル（pull） | リモート（GitHub）の記録をローカルリポジトリに取り込むこと |
| マージ | ブランチの変更をmainブランチに統合すること |
| プルリクエスト（PR） | ブランチの変更をmainブランチに取り込む申請（`git pull` とは別の概念） |
