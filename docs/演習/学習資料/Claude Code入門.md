## この資料の目的

このトレーニングでは、ほぼすべての課題でClaude Codeを使います。

「とりあえず使えれば、何でも答えてくれるのでは」と感じるかもしれません。しかし実際には、**どんなプロンプト（AIへの指示）を書くか**によって出力の質が大きく変わります。また、AIが出した答えを**どう扱うか**を知っていないと、誤った成果物を正しいものと勘違いしたまま進んでしまいます。

この資料の目的は、Claude Codeを最初から正しく使うための「地ならし」です。

---

## Part 1：ClaudeとClaude Codeとは

### Claudeとは

ClaudeはAnthropicが開発したAIアシスタントです。ChatGPTと同じ種類のもので、テキストで質問するとテキストで答えを返します。コード・文書・分析・推論など、得意なことの範囲もほぼ同じです。

「ClaudeにできてChatGPTにできない」という場面もありますが、このトレーニングでの使い方において本質的な違いはありません。今回はClaudeを使う、と覚えておけば十分です。

### Claude Codeとは

Claude CodeはVS Codeのターミナルから操作するタイプのツールで、Claudeを呼び出して使います。このようなツールをCLIツール（ターミナルから命令を入力して操作するツール）と呼びます。

一般的なAIチャット（ChatGPTのWeb版など）との最大の違いは、**ローカル環境のファイルの読み書きやコマンド実行と直接連携できる**点です。

```
一般的なAIチャット：
  あなた → テキスト → AI → テキスト → あなた

Claude Code：
  あなた → テキスト → AI → テキスト・ファイル操作・コマンド実行 → あなた
```

このトレーニングでは、テストコードの生成やファイル作成なども、Claude Codeを通じて行います。

### 起動と終了

VS Codeのターミナルで `claude` と入力してEnterを押すと起動します。終了は `/exit` またはCtrl+Cです。

```
> claude
```

## Part 2：壁打ち相手としてのClaude Code

### 「壁打ち」とは

このトレーニングではClaude Codeを「壁打ち相手」と表現しています。

壁打ちとは、テニスで壁に向かってボールを打ち続ける練習のことです。壁は考えてくれませんが、ボールを打てば必ず返ってきます。Claude Codeとの壁打ちはそれより少し賢く、こちらの問いに応じた返答が返ってきます。

ただし、壁は「正しい打ち方」を教えてくれるわけではありません。同じように、**Claude Codeは「正解を保証する存在」ではありません。** 返ってきたボール（AIの出力）を、あなたが判断して次に活かすのが壁打ちです。

### AIが得意なこと・苦手なこと

| AIが得意なこと | AIが苦手なこと |
|---|---|
| テストケースの案を大量に出す | 「これで十分か」を判断する |
| コードのテンプレートを素早く生成する | 仕様と照らして「正しいか」を検証する |
| 知識や概念をわかりやすく説明する | あなたの具体的な文脈を最初から把握する |
| 表・箇条書きなど指定した形式で出力する | 自分の出力に含まれる矛盾を自ら検出する |

AIは「それらしく見える出力を高速で生成する」ことが得意です。しかし「正しさの保証」はあなたが担います。

## Part 3：プロンプトのつくり方

### プロンプトとプロンプトエンジニアリング

AIへの指示や問いかけのことを**プロンプト**と呼びます。

プロンプトの書き方を工夫してAIの出力の質を上げることを**プロンプトエンジニアリング**と呼びます。「エンジニアリング」という語が使われていますが、特別なプログラミング技術の話ではありません。「どう指示すればより良い出力が返ってくるか」を意識して問いかけを組み立てること——それがプロンプトエンジニアリングです。

このPartで学ぶのは、プロンプトエンジニアリングの基本です。

### 文脈を与える

Claude Codeはあなたとの会話が始まった時点では、あなたが何をしているか・何を知っているかを把握していません。

「テストケースを作って」というプロンプトを送っても、何のシステムの・どの機能の・どの仕様に基づいた・どのレベルのテストケースなのかを知らなければ、的外れな出力になります。

文脈として意識するとよいのは次の3点です。

| 要素 | 具体例 |
|---|---|
| **誰として答えてほしいか** | 「テストエンジニアの立場で」 |
| **何の話をしているか** | 「会員登録システムの」「ログイン機能について」 |
| **何を求めているか** | 「境界値分析の観点でテスト条件を洗い出して」「初心者にもわかるように説明して」 |

### 悪いプロンプト vs 良いプロンプト

**悪い例：**
```
テストケースを作って
```

**良い例：**
```
テストエンジニアの立場で考えてください。
会員登録システムの年齢入力フィールドについて確認したいです。
仕様は「18歳以上を成人会員、18歳未満を未成年会員として扱う」です。
境界値分析の観点で、テスト条件を洗い出してください。
```

後者の方が的を射た出力が得られます。これは「AIが賢いかどうか」の話ではなく、「プロンプトに必要な情報が渡っているかどうか」の話です。

**注：** 良い例では「テストケースを作って」ではなく「テスト条件を洗い出してください」という指示にしています。テスト条件とテストケースの違いは課題1で詳しく学びます。

### 会話を積み重ねる

Claude Codeは起動中（セッション内）は会話の流れを記憶しています。一度に完璧な回答を求める必要はありません。

```
最初のプロンプト → 大まかな出力を得る
  ↓
「漏れていると思うケースを追加して」
  ↓
「このケースはなぜ必要だと思った？」
  ↓
自分の考えと照らして判断する
```

こうした往復が「壁打ち」の実態です。

## Part 4：出力との付き合い方

### 出力はたたき台

Claude Codeが出した内容は「たたき台」として扱ってください。

テストコードを生成したとき、コードが動くかどうかは実行して確認します。しかし注意が必要なのは、「**実行はパスするが、正しいことを確認していない**」ケースです。

たとえば「成人会員の料金が未成年会員より高いことを確認するテスト」を依頼したのに、生成されたコードが実際には「成人会員の料金が登録されていること」しか確認していない、ということが起こり得ます。

「実行がパスする」と「正しいテストである」は別の話です。

### 「なぜ？」を問い返す習慣

AIの出力を受け取ったときに、次の問いを持つ癖をつけてください。

- この出力は、自分が求めたものと一致しているか？
- このテストケースは、どの仕様に基づいているか？
- 「仕様書に書いていない仮定」が含まれていないか？

わからなければClaude Codeに「なぜそのケースが必要だと思った？」と問い返すことができます。理由を説明できない出力は、採用するかどうかを慎重に判断してください。

### AIは間違える

Claude Codeは、もっともらしく見える誤った情報を出すことがあります。

このトレーニングでは実際に、AIが生成したコードに誤りが含まれており、実行して初めて発覚するケースを体験します。「AIが言ったから正しい」という前提を持たないことが、このトレーニングを通じてまず身につけてほしいスタンスです。

## Part 5：このトレーニングでのClaude Codeの使いどころ

各課題でClaude Codeを使う場面は次のとおりです。

| 課題 | Claude Codeの使い方 |
|---|---|
| 課題1〜2 | テスト条件・テストケースの案を出してもらい、仕様と照らして検証する |
| 課題3 | Postmanの操作でつまずいた際に問いかける |
| 課題4 | APIのテストケースをClaude Codeと壁打ちしながら設計する |
| 課題5〜6 | テストモデル（状態遷移図・シーケンス図）のドラフトを生成してもらう |
| 課題7〜8 | Playwrightのテストコードを生成してもらい、実行して確認する |
| 課題9 | 課題1〜8の全要素を使い、テスト設計から不具合報告まで一気に進める |

いずれの課題も、「**Claude Codeが作ったものを、あなたが確認・修正して成果物にする**」という流れは変わりません。

---

## まとめ

Claude Codeは壁打ち相手です。答えは返ってきますが、正解を保証する存在ではありません。

1. **文脈を与えると、出力の質が上がる。** 「誰として・何の話か・何を求めるか」を整理してからプロンプトを送る。
2. **出力はたたき台。** 実行がパスすることと、正しいテストであることは別の話。
3. **問い返す習慣を持つ。** 「なぜそのケースが必要か」を説明できない出力は、採用するかどうかを慎重に判断する。

では、課題1に進みましょう。
